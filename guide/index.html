Fable is a studio showcase: twenty-five fictional products, each shipped as a complete, individually-designed website. This is the honest, end-to-end account of how they were made — and how you could reproduce the whole thing.

01The idea

Most “template galleries” show the same layout twenty-five times with the colours swapped. That teaches you nothing, because the hard part of design isn’t the grid — it’s fit: the way a wine club should feel unhurried and a transit app should feel instantly legible, and how that conviction reaches every type choice, easing curve and interaction.

So Fable started from concrete product concepts instead of abstract templates. A sleep wearable. An eVTOL air-taxi. A literary magazine. A citizen-astronomy network. Twenty-five briefs across twenty-five categories — each one forcing a different palette, voice, motion language and signature technique. A fictional product is a design constraint generator: it tells you what can’t be generic.

The through-line. What every site shares is not a look — it’s a rigor bar: design tokens in :root, a display+body+mono type system, a named signature easing, a clipped-line hero intro, IntersectionObserver scroll reveals, a full prefers-reduced-motion fallback, :focus-visible keyboard support, real alt text, and a per-site build guide. Same engineering discipline; twenty-five divergent identities.

02The workflow

Each site moved through the same pipeline, but every stage produced deliberately different output.

  1. Ideation

    Write the brief as a real product: what it is, who it’s for, its voice, and the one feeling the homepage must deliver in five seconds. That feeling becomes the site’s signature technique.

  2. Per-project Mobbin UX research

    For each category, study 5–10 shipped products via Mobbin — onboarding, navigation, empty/loading states, microcopy. Two or three real interaction patterns per site are adopted as the UX backbone, so the invented visuals sit on top of proven ergonomics rather than guesswork.

  3. Asset generation

    Seven sites needed a photographic hero (cabins, vineyard, surf, ceramics, coffee, plants). Those stills were generated with Higgsfield, then compressed. Everything else is procedural — WebGL, canvas and SVG generated at runtime: starfields, terrain, wireframes, molecular models, wave-fields, pixel dungeons. No stock, no bloat.

  4. Build — one identity at a time

    Each site gets its own palette, type trio, signature easing and signature technique. No two share a display+body+mono combination; no two share a technique. Vanilla sites are three files (index.html, styles.css, main.js); framework sites static-export.

  5. Three iteration passes

    Structure (layout, content, semantics) → Motion (hero intro, reveals, the signature moment, plus a working reduced-motion path) → Polish (contrast, focus states, a Lighthouse-minded performance pass, lazy-loading, weight budget).

  6. Deploy

    Every site ships to GitHub Pages as its own repo. Vanilla sites use relative paths and a .nojekyll flag; framework builds set a base path and publish the exported /docs (or dist) output. This hub links them all together.

03The tools

  • Claude (Fable) — the builder. Wrote the HTML, CSS, shaders and interaction code for all twenty-six sites, and this guide. The visual design decisions, procedural graphics and copy are AI-generated; that’s stated plainly here and in every site’s README.
  • Mobbin MCP — UX research: real patterns from shipped products, per category.
  • Higgsfield — the seven photographic hero stills (the only non-procedural imagery).
  • GitHub CLI + Pages — hosting; one public repo per site, Pages served from main.
  • Playwright — headless Chromium to capture the 1600×1200 gallery thumbnails you see on the hub.

Honest about what’s AI-generated. These are design-showcase concepts, not shipping products. Interactive “engines” (a RAW developer, a robo-advisor projection, a molecular viewer) are faithful browser-side approximations built to demonstrate the design — not production backends. Each site’s README carries a demo-vs-real map.

05Awwwards notes

All twenty-five are submission-ready — each has a 1600×1200 thumbnail, a reduced-motion path and keyboard support. If you were submitting a subset first, this is the recommended priority and why:

  1. Perigee — strongest single visual + interactivity

    A 72k-star GPU shader starfield feeding a live, draggable sky console. The most immediately arresting hero of the set, and it’s genuinely interactive.

  2. Hangar 12 — procedural craft + fallback engineering

    A procedurally-built eVTOL wireframe with animated spec callouts, and a clean static fallback — judges reward technique that degrades gracefully.

  3. The Marginalia — editorial innovation

    Scroll-synced marginalia and drop caps make a reading experience feel designed, not templated. Stands out in a field of product landing pages.

  4. Static & Noise — synthesized, audio-reactive

    A Web Audio visualizer reacting to a loop synthesized in the browser — sound design is rare in submissions and memorable to juries.

  5. Fieldnote — a working product demo

    A real ⌘K command palette and citation hover-cards. It doesn’t just describe a tool; it lets you use one.

Awwwards recommends a short video pass per submission; that’s left as a manual step here, since it needs an account and payment.

06Reproduce it

To run the whole thing end to end:

  1. Write twenty-five product briefs across distinct categories; for each, name the one homepage feeling and its signature technique.
  2. Research each category on Mobbin; adopt 2–3 real interaction patterns per site.
  3. Generate only the photographic heroes; build everything else procedurally (WebGL/canvas/SVG).
  4. Build each site to the shared rigor bar with a unique palette / type trio / easing / technique.
  5. Run Structure → Motion → Polish, including a reduced-motion path and a weight budget.
  6. Ship each as its own GitHub Pages repo; capture a thumbnail with Playwright; link them from a hub.
# a vanilla site, start to shipped
gh repo create bswxyz/<slug> --public --source . --push
gh api --method POST /repos/bswxyz/<slug>/pages \
  -f 'source[branch]=main' -f 'source[path]=/'
gh repo edit bswxyz/<slug> --homepage "https://bswxyz.github.io/<slug>/"
# live in ~1 min; .nojekyll keeps Pages off the folder structure
